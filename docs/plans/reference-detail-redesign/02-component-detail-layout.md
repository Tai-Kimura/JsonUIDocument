# Plan 02 — Component detail layout (`reference/components/<name>.json`)

## Why

Today's component page is "breadcrumb → title → description → usage → attributes list → examples list → related pills → next reads" — one vertical stack, no anchors, no canonical preview, no grouping of 40+ attributes. This plan is the new container structure.

The template lives inside [scripts/build-attribute-reference.ts](../../jsonui-doc-web/scripts/build-attribute-reference.ts) (the `writeComponentLayout` function, or equivalent — confirm at implementation time). The 28 existing component pages are regenerated by `npm run build:attrs` after the template changes.

## In scope

- The full left-column content structure below the shared page header.
- New data-block entries the layout needs.
- The grouping / tab UX for the attribute list (there are 9 attribute categories today — we display a sticky mini-nav so 40-attribute components become navigable).
- The per-example platform tab UX (delegated to Plan 05 for the cell itself; this plan specifies how the page owns the tab state if needed).
- The 2-column wrapper with right-rail TOC (structure from Plan 01; anchor list authoring is Plan 02's job).

## Out of scope

- The shared header (Plan 01).
- The attribute row cell (Plan 04).
- The code example cell (Plan 05).
- The override schema and generator changes (Plan 06).
- Attribute category pages (Plan 03).

## Content structure — detailed

Each top-level section in the body column has a leading small-caps eyebrow label (uppercase, 11px, `#64748B`, letter-spacing 0.08em) and a 1px rule below it. Makes the page scannable without heavy h2 typography.

### § 3 — Attributes

Eyebrow "ATTRIBUTES", then a horizontal chip list of category names (driven by a CollectionDataSource seeded from the server-rendered list of categories present on this component — a component that has no Binding attrs doesn't show the Binding chip). Each chip is a Button whose `onClick` calls `router.replace('#section-<category>')` — browser native anchor scroll.

Then **N group sections**, one per category present on this component. Each group:

```
  <Label "Common" 18px bold, id="section-common">
  <Collection attributes_common cell="cells/reference_attribute_row" ... items="@{attributesCommon}">
```

The data block now carries **one CollectionDataSource per category** rather than a single flat `attributes` list:

```json
{ "name": "attributeCategories", "class": "CollectionDataSource" }   // the chip list
{ "name": "attributesCommon", "class": "CollectionDataSource" }
{ "name": "attributesStyle", "class": "CollectionDataSource" }
{ "name": "attributesLayout", "class": "CollectionDataSource" }
{ "name": "attributesSpacing", "class": "CollectionDataSource" }
{ "name": "attributesAlignment", "class": "CollectionDataSource" }
{ "name": "attributesBinding", "class": "CollectionDataSource" }
{ "name": "attributesEvent", "class": "CollectionDataSource" }
{ "name": "attributesState", "class": "CollectionDataSource" }
{ "name": "attributesMisc", "class": "CollectionDataSource" }
```

The nine category CollectionDataSources match [docs/data/attribute-categories.json](../../docs/data/attribute-categories.json). Empty categories (no attributes) resolve to an empty CollectionDataSource; their group section's `visibility` is `gone` (computed in the ViewModel).

### § 4 — Tabs UX for the attribute category chips

**Decision (v1):** the chips don't toggle visibility — all category sections are rendered unconditionally. The chips are anchor navigators (`onClick` pushes `#section-<category>` to the URL hash). Browsers' native scroll-to-anchor handles the jump.

**Why not real tabs:** testing `TabView` inside a matching-parent-height non-scrolling column is brittle today (`TabView` expects a scroll owner). Anchor nav is simpler, Cmd-F-friendly (everything is always on the page), and printable.

If we revisit this later, the chip Collection's cell becomes a Button that writes to a `@{activeCategory}` uiVariable, and each group's `visibility` becomes `@{activeCategory == 'common' || activeCategory == 'all' ? 'visible' : 'gone'}`. We don't implement that in v1.

### § 5 — Examples

Eyebrow "EXAMPLES", then a Collection that renders each example. The cell (Plan 05) owns the per-example tab UX — this layout only passes it the data.

Data shape stays the same CollectionDataSource wrapper; each cell receives one row with N platform-variant code strings. Plan 06 extends the override schema to support per-example platform variants; Plan 05 specifies the cell.

Anchor IDs: each cell's root View has `id: "example_<n>"` (mapped to `#example-1`, `#example-2` via the DOM id). The cell emits this from the `id` field in the data row.

## TOC rail contents

The right-rail `TableOfContents` gets one entry per category group (Common, Style, Layout, ...) plus one entry for "Examples" and one for "Related components". Top entry is "At a glance". Total ~10 entries. Seeded in the ViewModel from the category presence flags.

## ViewModel work

Existing hook [useComponentReference.ts](../../jsonui-doc-web/src/hooks/reference/useComponentReference.ts) needs:

1. Split the single `attributes` CollectionDataSource into **9 category sources**. The build script (Plan 06) emits the split shape in the runtime JSON; the hook just wraps each.
2. Compute 10 `visibility` bindings: one per category group + usage + alias callout.
3. Compute the stat strings (`statRequired`, `statOptional`, `statEvents`) or accept them from the runtime JSON. Recommend emitting from the build script so the hook stays thin.
4. Seed `tocEntries` CollectionDataSource (if the `TableOfContents` custom component takes a data source — confirm its contract).
5. Wire `onCopyTypeName` to write `"type": "${component}"` to clipboard.
6. Wire `onNavigateBreadcrumb` to `router.push('/reference/components')` — replaces today's `onNavigateBack`.

## Validation

1. `jui build` zero warnings.
2. `jui verify --fail-on-diff` clean.
3. `tsc` clean.
4. Rendered Button page at 1440px: canonical CodeBlock shows the `"Text-only"` example by default, stat chips show realistic counts (e.g. `6 required · 38 optional · 4 events`), category chip row shows 6-9 chips, clicking "Style" jumps to `#section-style`, each attribute row renders as specified in Plan 04.
5. Rendered Button page with `usage` omitted (test via a component that has no `usage` in its override): the USAGE section is entirely gone (no empty gap).
6. Right rail TOC is sticky and stays in the viewport on scroll; clicking an entry jumps to the matching `id`.
7. At 1024px: right rail collapses, content goes full-width, category chips wrap.
8. At 768px (for completeness): layout still readable; no horizontal scrollbar.

## Open questions

- **Usage vs at-a-glance order.** Is "at a glance" (code preview) better above or below "usage" prose? Recommend code-first (land on the snippet, scan the prose only if needed). Revisit if user testing shows otherwise.
- **Category order.** Categories should sort the same way across every component page. Recommend: Common → Style → Layout → Spacing → Alignment → State → Binding → Event → Misc. Hand-authored in the build script.
- **"Copy type" vs "Copy JSON".** The header's copy button currently writes `"type": "Button"`. Should the AT A GLANCE's canonical code also have its own copy button? CodeBlock already has `copyable: true` so yes, implicitly. Good enough.
